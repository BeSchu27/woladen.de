## Main Finding

Station coverage alone is not enough for provider-to-provider analysis. Some providers have high `mapped_stations_observed / static_matched_station_count` but still a very poor mapped-observation ratio, which means the current matching layer only recognizes a small share of their live EVSE observations.

Use two filters before calling anything “competitive analysis”:

- station coverage against static matches
- mapped observation ratio from `provider_daily_summary.csv`

The scorecard now codifies this directly:

- `competitive_analysis_eligible = 1` only when there are static matches, parseable messages, and `mapped_observation_ratio >= 0.5`
- `competitive_analysis_tier = review` when `0.2 <= mapped_observation_ratio < 0.5`
- all lower-quality or structurally incomplete providers land in `competitive_analysis_tier = exclude`

## Initial Matching Diagnosis

- `qwello`: low mapping quality is not caused by a simple ID-format mismatch. The existing matched EVSE IDs already use the same UUID-like format as the unmapped samples, so the problem is sparse static/match coverage.
- `wirelane`: same pattern as `qwello`. Existing matches already use hashed UUID-style identifiers, but the live archive emits many additional IDs that are not present in the static match inventory.
- `m8mit`: site matches exist, but dynamic samples are mostly AFIR-style EVSE IDs such as `DEMSUESWBEB42`, while current matched EVSE IDs are a mix of numeric IDs, hashes, and a smaller AFIR-style subset. This looks like a cross-feed identifier reconciliation problem.
- `mobidata_bw_datex`: zero static site matches and zero matched EVSE IDs explain the `0.0` mapping ratio. This provider should stay out of competitive comparisons until static mapping exists.
- `deprecated_chargecloud`: zero static matches suggest this feed should either remain excluded or be reconciled explicitly against the active `chargecloud` provider inventory.
